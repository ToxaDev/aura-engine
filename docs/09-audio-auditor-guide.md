# AuraEngine Signal Path: Step-by-Step Auditor's Guide

This document is a comprehensive walkthrough of the DSP pipeline inside the AuraEngine converter. It is intended for code auditors, mathematicians, and developers who need to trace every micro-modification applied to an audio signal — from the original file through to the output FLAC.

All paths below are relative to `desktop-app/src-tauri/src/audio/`.

> ⚠️ **Corrections from the 2026-07 audit** (see [13-pipeline-hardening-2026-07.md](13-pipeline-hardening-2026-07.md)):
> - **True-Peak ceiling = −0.5 dBTP** (linear ≈ 0.9441), not `1.00116`.
>   Gain is reduced only when the ceiling is exceeded; quieter files pass through bit-exact.
> - **DC blocking is always applied** (no `1e-6` threshold): either mean subtraction across the
>   full channel or a 2 Hz IIR HPF (`iir_dc_blocking`).
> - **Noise shaping is disabled at output rates **> 48 kHz** (not 96 kHz)**:
>   all hi-res outputs receive flat TPDF only.
> - **CPU OLA block = 32768**, fixed. The 262144–2097152 range applies to the GPU path only.
>   Algorithmic latency: CPU 2×b\_size, GPU 1×b\_size.
> - `bessel_i0` is a **live** helper used by the Kaiser window in `apodize.rs` and
>   `gpu/fft_math.rs` — it is not dead code.
> - Hybrid-Phase switching is **stereo-linked** (decision based on the mid signal);
>   dither is **independent per channel**; the polyphase path also dithers and verifies.

## Stage 1: Lossless Decoding (Extraction)

**File:** `converter/decode.rs`
**Function:** `decode_file()`

Captures the original waveform from disk (FLAC, WAV, etc.).

- Uses the `symphonia` library.
- To avoid losing the 24th mantissa bit on 24-bit files, the decoder requests an integer buffer `SampleBuffer::<i32>`. Symphonia left-shifts any integer format to full 32-bit boundaries.
- A 64-bit (f64) audio buffer — `audio.samples_l` and `audio.samples_r` — is formed by dividing the integer samples by the exact constant 2^31 (`1.0 / 2_147_483_648.0`).
- Gapless trim is applied: encoder-delay and padding frames (`codec_params.delay` / `padding`) are stripped so that LAME priming silence never enters the FIR chain.

---

## Stage 2: DC Blocking

**File:** `converter/process.rs`
**Function:** `prepare_audio_phase()`

Protects the large FIR filter from low-frequency energy bias.

- DC blocking is **always applied** — there is no threshold below which it is skipped.
- The implementation is either mean subtraction across the full track (left and right channels independently) or a 2 Hz IIR high-pass filter (`iir_dc_blocking`), depending on the source.
- This guarantees a true silence baseline and prevents the filter from producing sustained low-frequency ringing.

---

## Stage 3: Headroom Pre-DSP

**File:** `converter/process.rs`
**Function:** `prepare_audio_phase()` (block following DC blocking)

Reduces signal level before the computationally intensive stages.

- If the user has set `headroom_db < 0.0` (e.g. −3.0 dB) in the UI, a gain multiplier `gain = 10^(headroom / 20)` is computed and applied to every sample.
- This headroom gives the 64-bit mantissa room to absorb energy peaks generated by the large FIR filter, preventing clipping in the internal f64 domain.

---

## Stage 4: Adaptive Apodizing v3 — Source Forensics

**Files:** `converter/pipeline/prepare.rs`, `converter/apodize.rs`
**Functions:** `analyze_source()`, `decide_apodizer()`, `apply_custom_apodizing()`

Treats source files for brick-wall filter artifacts baked into the recording
chain — at any container rate (full spec: [doc 14](14-adaptive-apodizer-v3.md)).

- **Spectral-cliff detector** (Welch long-term spectrum): a brick-wall SRC/ADC
  leaves tens of dB dropped within a fraction of an octave with only a noise
  floor above. A cliff well below a hi-res container's Nyquist unmasks fake
  hi-res (an upsampled 44.1/48 kHz master) — all further analysis then runs
  against the *original* Nyquist.
- **Time-domain pre-ring detector**: a 127-tap Kaiser high-pass at
  0.78×(effective) Nyquist isolates the near-Nyquist band; the strongest
  broadband attacks across the whole track are checked for a high-frequency
  burst *just before* each attack — the fingerprint of a linear-phase brick
  wall. The verdict is based on the fraction of ringing attacks.
- **The ring frequency is measured** (per-burst FFT, median across attacks
  with an agreement gate): the corrective minimum-phase Kaiser lowpass lands
  just below the measured filter edge instead of a preset bucket. Severity
  selects the depth (β=24 strong / β=14 mild); taps scale with the container
  rate (4096…32768).
- **Mirror-image alias probe**: a bad upstream resampler leaves images of the
  content above the original Nyquist (a tone at f gets a twin at 2·Ny−f). The
  detrended spectral shape just below each candidate legacy Nyquist is
  compared, segment by segment, with the mirrored band above it — images
  correlate bin-for-bin, honest hi-res content never does. Detected images
  are cut regardless of the pre-ring verdict.
- **Conservative by design**: a cliff without pre-ring on transient-rich
  material means the source filter is minimum-phase or the master was already
  apodized — logged as a diagnosis, audio untouched. No actionable evidence →
  the user's static preset (or nothing at all).
- The `AA`/`Apod` output filename tag reflects the apodizing that **actually
  ran** — files that received a clean verdict are not mislabelled.

## Stage 5: Resampling (Sinc Interpolation)

**File:** `converter/process.rs`
**Location:** `SincFixedIn::<f64>::new(...)` block inside `process_one_prepared()`

Brings the signal to the target sample-rate grid.

- Active only when `source_rate != target_rate` (e.g. 44.1 → 384 kHz). Uses the `rubato` library.
- Configuration: `SincInterpolationType::Cubic` (for sub-sample accuracy) with a `BlackmanHarris2` window.
- Streams data through in 4096-sample f64 chunks.
- After this stage, aliasing mirrors from non-integer up-sampling ratios may still be present; they are removed by the main FIR convolution in Stage 6.

---

## Stage 6: Main 64-bit Convolution (128-bit Coefficients)

**Files:** `converter/process.rs` → `dsp_core.rs` (CPU) or `gpu/processor.rs` (Vulkan)
**Entry points:** `new_with_coefficients()`, `process_audio()`

The core reconstruction step: ideal band-limiting using a precomputed, near-perfect sinc filter.

- Filter coefficients are loaded from a `.npy` file produced by the offline Python filter-design tool (located outside this repository). The coefficients were computed with **128-bit (quad-precision) arithmetic** using `mpmath`, including Kahan-normalized sinc values, then saved for use inside the engine's **64-bit f64 convolution**.
- The FIR filter can contain up to ~30 million taps, depending on the selected preset (1M / 5M / 10M / 30M — these tags match the filter-file names exactly).
- Convolution uses **uniform-partitioned overlap-add (OLA)**: the large filter is split into ~900 micro-blocks, and each FFT step operates on a window of `32768 × 2 = 65536` samples. This is the fastest known algorithm for real-time convolution of very long filters.
- The **CPU OLA block size is fixed at 32768 samples**. The 262144–2097152 block-size range applies only to the GPU path.
- Algorithmic latency: **CPU path = 2 × b_size** (65536 samples), **GPU path = 1 × b_size** (32768 samples). The pipeline trim and flush arithmetic queries `output_latency()` on the `DspProcessor` trait and does not hardcode a block size.
- After convolution the FIR tail is flushed, and the group delay `(taps − 1) / 2` is compensated by trimming the corresponding number of leading samples from the output.

## Stage 7: Hybrid-Phase Engine

**Files:** `converter/process.rs`, `hybrid_phase.rs`, `hpss_native.rs`

Combines the phase-linearity of a linear-phase FIR with the transient sharpness of a minimum-phase FIR.

1. **Minimum-phase convolution** (`converter/process.rs`): the audio stream is processed a **second time** through a precomputed minimum-phase version of the same large FIR filter (derived via cepstral inversion with DC renormalization). This produces a copy of the track with no pre-ringing but a frequency-dependent group delay.
2. **Transient analysis** (`hpss_native.rs`): the dry signal is analysed via an f64 spectrogram. Median filters separate percussive content from harmonic content (HPSS). An ~86 Hz onset-energy envelope is computed for the entire track using native Rust, **before streaming begins**, and upsampled to the output rate with Catmull-Rom (C1-continuous) interpolation. This envelope is stored in a compact `O(duration × 100 Hz) float64` array (approximately 240 KB for a 5-minute track), providing mathematically unlimited look-ahead without holding a large audio preview buffer in RAM. The `.onset_envelope.json` cache is version-tagged (`hpss_native_rust_v4`) so stale caches from earlier detector versions are automatically regenerated.
3. **Stereo-linked blending** (`hybrid_phase.rs`): the switch between linear-phase and minimum-phase is computed from the **mid signal** `((dL + dR) / 2)` and applied identically to both channels. This means L and R always switch at the same sample, eliminating the inter-channel timing artifacts (up to ±5 ms) that occurred when each channel switched independently. The mid signal is used for analysis only — no M/S transform is applied to the audio itself; each channel's output is still its own linear-phase or minimum-phase sample.

   The blend threshold is the onset-envelope value; at low-energy moments (silence, sustained tones) 100% linear-phase is used, and the signal fades into minimum-phase approximately 15 ms before a detected transient.

## Stage 8: True Peak Inter-Sample Protection

**File:** `converter/dsp/true_peak.rs` (called from `converter/process.rs`)
**Function:** `apply_true_peak_normalization()` — runs **strictly before dithering**

Prevents inter-sample clipping in downstream DACs.

- Digital sample values represent discrete points on a continuous waveform. The true analogue peak can occur *between* two digital samples and exceed the largest coded value.
- A **4× oversampling pass** using Polyphase Lanczos-4 sinc interpolation is applied to **every sample in the file** (a full track pass, conforming to ITU-R BS.1770).
- The **True-Peak ceiling is −0.5 dBTP** (linear ≈ 0.9441). If the computed true peak exceeds this ceiling, the entire file is scaled down by the necessary amount — a pure gain change with no compression. If the true peak is below the ceiling, this stage is a bypass and the file's dynamics are unchanged.

## Stage 9: Dithering and Noise Shaping

**File:** `converter/dsp/dither.rs` (called from `converter/process.rs`)
**Function:** `apply_dithering_and_noise_shaping()` — the final DSP step before encoding

Converts the internal 64-bit float representation to 24-bit integer, minimising quantization distortion.

- The entire engine processes audio in 64-bit float. Output is quantized to 24-bit (q_step = `1.0 / 2^23`). Simple truncation of the lower bits produces hard harmonic distortion; dithering replaces that with inaudible noise.
- **TPDF dither**: two independent PRNG streams generate a triangularly distributed white noise signal, one per channel. Left and right channels draw from **separate** `SmallRng` streams (`dither.rs`) so their dither floors are uncorrelated.
- **9th-order Wannamaker noise shaping** computes the rounding error between the 64-bit value and its 24-bit approximation and feeds it back through a psychoacoustic filter that pushes the shaped error above 15–20 kHz, where hearing is least sensitive.
- **Noise shaping is disabled at output sample rates above 48 kHz.** At high rates the Wannamaker coefficients would push the shaped error to extreme ultrasonic frequencies (e.g. ~130 kHz at 384 kHz output) with +40 dB amplitude — potentially harmful to tweeters and measurement equipment. At all hi-res output rates (88.2, 96, 176.4, 192, 352.8, 384 kHz) the engine applies flat TPDF dither only.
- After the shaper/dither feedback, samples are clamped to ±(1 − q_step), so a dither excursion can never exceed the largest representable 24-bit code. The error feedback still uses the unclamped value so the shaper remains stable.

---

## Stage 10: FLAC Encoding

**File:** `converter/encode.rs`
**Function:** `encode_flac()`

Lossless packaging of the processed signal.

- The 64-bit L and R sample streams are interleaved.
- The interleaved buffer is written as raw `f64le` (64-bit float, little-endian) to a `std::process::Stdio` pipe.
- `FFmpeg` reads the pipe and encodes to 24-bit FLAC (`-c:a flac -sample_fmt s32`).
- **FFmpeg performs no DSP of its own.** It receives a signal that has already been dithered and quantized by the engine and acts purely as a container formatter. If conversion is cancelled or fails, the partial output file is deleted — no corrupt FLAC is left behind.

---

## What is not in the standard signal path

**Polyphase FIR Resampling path** (`use_fir_resampling = true`): an alternative path in which the FIR filter also functions as the resampler (no `rubato` in the chain). This path resolves filters through `find_precomputed_filter`, runs the L sub-filters in parallel on CPU (rayon) or sequentially on GPU via `run_polyphase_pass`, and applies the same dither and bit-perfect FLAC verification as the standard path. It is exposed by the **Polyphase FIR Resampling** checkbox in Advanced DSP settings. Because the output length is exactly `input × L` by construction, this path is also free of the ~0.4 s of trailing zero-padding that the standard (rubato) path leaves at the end of each file. The checkbox is locked during an active conversion.

**XTC** (`dsp/xtc.rs`): a complete transaural crosstalk canceller. It is mathematically honest but compensates for listening geometry rather than anything in the recording, so it is intentionally left as dead code and unwired from the pipeline.

**`bessel_i0`** (`apodize.rs`, `gpu/fft_math.rs`): a live helper function used to evaluate the Kaiser window in the apodizer and in the GPU FFT math module. It is **not** dead code.

**OLA block-size note**: if you see a 32768-sample block size alongside a 30-million-tap filter count in the logs, this is expected and correct. Uniform-partitioned OLA splits the giant filter into ~900 micro-blocks; the effective FFT size per step is `32768 × 2 = 65536`. The ratio is not a misconfiguration.

**Offline filter design tool**: filter coefficients are generated by a Python tool using `mpmath` quad-precision arithmetic. That tooling is maintained separately and is not included in this repository.
